# Pre-Implementation Completion Checklist

This document records closure of the Phase 2 pre-implementation gaps and maps
each previously open item to the contract now frozen in:

- `DETAILED_PLANNING_PHASE_2.md`
- `ADRs_PHASE_2.md`

Its purpose is now slightly different from the original draft:

> show that Phase 2 is specified tightly enough to begin broad implementation
> for the documented contract, while keeping deliberate deferrals visible.

## Current Readiness Verdict

Phase 2 is now implementation-ready for the frozen contract.

The remaining open work is implementation work, not contract-definition work.
Deliberately deferred items still exist, but they are now explicit Phase 2
trade-offs rather than blockers.

## Checklist Closure Details

### 1. Backend Selection Contract

Status: `Closed`

Closure decision:

- Phase 2 supports exactly two backend modes: `state_vector` and
  `density_matrix`.
- `state_vector` remains the default.
- `density_matrix` must be selected explicitly for exact noisy mixed-state
  claims.
- Unsupported density workflows fail early with a hard error.
- Silent fallback and implicit `auto` behavior are explicitly excluded.

Trade-off recorded:

- Backward compatibility is preserved, but users lose the convenience of
  ambiguous auto-selection.
- This is intentional because benchmark and paper claims must name the backend
  unambiguously.

### 2. Observable Contract for `Tr(H*rho)`

Status: `Closed`

Closure decision:

- The core Phase 2 observable is exact real-valued Hermitian energy evaluation
  through `Re Tr(H*rho)`.
- The required interface is the existing VQE sparse-Hamiltonian path.
- The canonical benchmark family is the XXZ Hamiltonian with local `Z` field.
- Generic non-Hermitian observables, POVMs, and batched multi-observable APIs
  are out of the core Phase 2 contract.

Trade-off recorded:

- The observable scope is scientifically strong and easy to validate.
- A broader general-purpose measurement API is deferred rather than
  half-specified.

### 3. Circuit-to-Density Bridge Contract

Status: `Closed`

Closure decision:

- The guaranteed Phase 2 bridge is the default `HEA` circuit generated by
  `qgd_Variational_Quantum_Eigensolver_Base`.
- The target density representation is `NoisyCircuit` plus `GateOperation` and
  ordered `NoiseOperation` insertion.
- User-supplied `qgd_Circuit` or `Gates_block` inputs are only optional Phase 2
  extensions when every operation lowers cleanly to the required gate set.
- Unsupported operations hard-error before execution.

Trade-off recorded:

- The bridge is partial by design, which keeps implementation bounded around a
  real workflow.
- Full `qgd_Circuit` parity is deliberately deferred.

### 4. Support Matrix

Status: `Closed`

Closure decision:

- Required gates: `U3`, `CNOT`.
- Required noise models: local single-qubit depolarizing, local amplitude
  damping, local phase damping or dephasing.
- Optional items remain clearly marked optional, including whole-register
  depolarizing as a regression or stress-test baseline and additional
  `NoisyCircuit` gates only when tests or microbenchmarks genuinely need them.
- Correlated noise, readout noise as a density-backend feature,
  calibration-aware noise, non-Markovian noise, and full gate parity are
  deferred.

Trade-off recorded:

- The support surface is narrow enough to be implementable.
- It still requires adding local depolarizing support, which is extra work, but
  that choice keeps the phase aligned with realistic local-noise studies.

### 5. Workflow Anchor

Status: `Closed`

Closure decision:

- The Phase 2 anchor workflow is noisy VQE ground-state estimation for a 1D XXZ
  spin chain with local `Z` field using the default `HEA` ansatz.
- The workflow contract requires:
  - full end-to-end workflow execution, including at least one reproducible
    optimization trace, at 4 and 6 qubits,
  - benchmark-ready fixed-parameter evaluation at 8 and 10 qubits,
  - and 10 qubits as the acceptance anchor for the exact regime.

Trade-off recorded:

- A single anchor makes the phase publishable and testable.
- It reduces generality, but that is preferable to claiming several vague
  workflow families without one fully supported one.

### 6. Benchmark Minimum

Status: `Closed`

Closure decision:

- Qiskit Aer density-matrix simulation is the required external baseline.
- The minimum benchmark set includes:
  - 1 to 3 qubit micro-validation circuits for each required gate and noise
    contract,
  - XXZ noisy VQE workflow cases at 4, 6, 8, and 10 qubits,
  - at least 10 fixed parameter vectors per mandatory workflow size,
  - one reproducible 4- or 6-qubit optimization trace,
  - and a full reproducibility bundle.

Trade-off recorded:

- This is strong enough for Paper 1 and strong enough to guide implementation.
- It intentionally stops short of a broad multi-simulator bake-off or a Phase 3
  performance campaign.

### 7. Numeric Acceptance Thresholds

Status: `Closed`

Closure decision:

- Mandatory acceptance coverage is 4, 6, 8, and 10 qubits.
- Maximum absolute energy error is `<= 1e-10` on required microcases and
  `<= 1e-8` on required workflow parameter sweeps.
- `rho.is_valid(tol=1e-10)` and `|Tr(rho) - 1| <= 1e-10` are mandatory on
  recorded validation outputs.
- `|Im Tr(H*rho)| <= 1e-10` is required for the exact observable path.
- Mandatory benchmark sets require a `100%` pass rate.
- A documented 10-qubit anchor case and full reproducibility bundle are part of
  the acceptance gate.

Trade-off recorded:

- The thresholds are strict enough to make Paper 1 defensible.
- No runtime speed threshold is included, because Phase 2 is a correctness and
  workflow-integration phase, not an acceleration phase.

## Resolution Outcome

The original recommendation was to close the checklist by adding a few companion
specs. That closure has now been performed directly inside the Phase 2 planning
and ADR documents instead.

This is better for implementation because:

- the source-of-truth planning document now contains the frozen contract,
- the ADRs now capture the reasons and trade-offs behind each boundary,
- and the checklist can point to accepted decisions rather than future TODOs.

## Go / No-Go Rule

Implementation can now begin safely in both modes:

- `Controlled early implementation`: allowed.
- `Broad phase implementation`: now also allowed for the frozen Phase 2
  contract.

Important interpretation:

- Any request outside the documented support matrix should now be handled as a
  new scope decision or follow-up ADR, not as implicit unfinished Phase 2
  definition work.

## Final Practical Assessment

The checklist is solidly closed.

Phase 2 is now specified tightly enough to start broad implementation for the
documented contract, with the main remaining uncertainty shifted from
specification to execution. The deferred items that remain are explicit future
work, not hidden blockers.
