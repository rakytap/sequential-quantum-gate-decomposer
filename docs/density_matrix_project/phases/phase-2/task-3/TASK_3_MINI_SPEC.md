# Task 3: Circuit-to-Density Backend Bridge

This mini-spec turns Phase 2 Task 3 into an implementation-ready contract. It
inherits the frozen phase decisions from `DETAILED_PLANNING_PHASE_2.md`,
`P2-ADR-011`, `P2-ADR-012`, and `P2-ADR-013`, plus the closed
circuit-to-density-bridge and support-matrix items in
`PRE_IMPLEMENTATION_COMPLETION_CHECKLIST.md`. It does not reopen backend-mode,
observable-scope, full-gate-parity, or workload-anchor decisions.

## Given / When / Then
- Given explicit `density_matrix` backend selection, the anchor noisy VQE
  workflow, and a circuit source within the frozen Phase 2 bridge surface.
- When the workflow lowers that circuit into the density-matrix execution path
  with explicit ordered local noise insertion.
- Then the request reaches `NoisyCircuit` through the documented HEA-first
  bridge, preserves the auditable operation order needed for exact noisy
  execution, and either executes on the supported path or fails before execution
  with a deterministic error naming the first unsupported bridge, gate, or
  noise condition.

## Assumptions and dependencies
- Task 1 provides explicit backend selection and no-fallback behavior for
  `density_matrix`.
- Task 2 and Task 4 consume the output of this bridge; this mini-spec defines
  lowering and routing behavior, not observable mathematics or the scientific
  choice of noise models.
- The guaranteed source representation is the default `HEA` circuit generated by
  `qgd_Variational_Quantum_Eigensolver_Base.Generate_Circuit()`.
- The guaranteed target representation is `NoisyCircuit` plus `GateOperation`
  and explicit ordered `NoiseOperation` insertion.
- The frozen required gate and noise surface comes from `P2-ADR-012`: `U3`,
  `CNOT`, and local single-qubit depolarizing, local amplitude damping, and
  local phase damping / dephasing.
- User-supplied `qgd_Circuit` or `Gates_block` inputs may appear only as
  optional extensions when every operation lowers cleanly to the documented
  required gate families; full `qgd_Circuit` parity is not assumed.
- This task defines the bridge needed for the anchor XXZ noisy VQE workflow; it
  does not define decomposition, partitioning, fusion, gradient, or
  acceleration behavior.

## Required behavior
- The bridge is a documented VQE-facing circuit-to-density contract rather than
  an implicit internal path.
- The required positive path begins from
  `qgd_Variational_Quantum_Eigensolver_Base.Generate_Circuit()` for the default
  `HEA` ansatz and lowers that circuit into the density execution path.
- The lowering target for supported requests is `NoisyCircuit` populated with
  the bridge-compatible gate sequence and explicit ordered local noise
  insertion.
- Bridge behavior must preserve the operation order relevant to exact noisy
  execution; noise insertion order must be explicit and reproducible rather than
  hidden or heuristic.
- The guaranteed supported gate families for the core Phase 2 path are `U3` and
  `CNOT`; the guaranteed supported noise models are the required local noise
  models frozen at the phase level.
- The bridge must be sufficient for the anchor XXZ noisy VQE workflow at the
  documented exact-regime sizes and must not require manual circuit rewriting
  beyond explicit noise specification.
- If a circuit source, lowered gate, fused block, or noise insertion falls
  outside the frozen support matrix, the request must fail deterministically
  before execution.
- The required failure path names the first unsupported bridge, gate, or noise
  condition so the unsupported case is auditable and reproducible.
- Any optional lowering of user-supplied `qgd_Circuit` or `Gates_block` inputs
  must obey the same hard-error unsupported behavior and must not widen the
  guaranteed Phase 2 contract beyond the documented support matrix.
- Validation, benchmark, and workflow artifacts for supported cases must make
  the circuit source and bridge path explicit so backend-integration claims
  remain auditable.

## Unsupported behavior
- Full `qgd_Circuit` parity or arbitrary `Gates_block` lowering as part of the
  core Phase 2 contract.
- Silent partial lowering, silent gate rewriting, silent omission of unsupported
  operations, or silent backend fallback when a request is outside the bridge
  surface.
- Treating `HEA_ZYZ`, binary-imported gate lists, or custom circuit sources as
  guaranteed bridge inputs for the mandatory Phase 2 path.
- Routing unsupported requests into a standalone density-only execution surface
  while still claiming the documented VQE bridge contract was satisfied.
- Injecting unsupported noise models, correlated noise, readout noise, or
  heuristic noise placement into the mandatory bridge behavior.
- Using Task 3 to imply support for decomposition flows, partitioning,
  gate-fusion compatibility, or general manual-circuit equivalence beyond the
  frozen anchor workflow.
- Returning ambiguous unsupported-case failures that do not identify the first
  unsupported bridge condition.

## Acceptance evidence
- Positive integration tests show that explicit `density_matrix` selection on
  the default `HEA` noisy VQE workflow lowers into `NoisyCircuit` with ordered
  `GateOperation` and `NoiseOperation` content.
- Support-matrix validation covers 1 to 3 qubit cases that exercise mandatory
  `U3` and `CNOT` lowering with each required local noise model individually and
  in mixed sequences.
- Workflow-level evidence shows that the bridge supports the anchor XXZ noisy
  VQE path for fixed-parameter evaluations at 4, 6, 8, and 10 qubits.
- At least one reproducible 4- or 6-qubit optimization trace exercises the full
  backend-selection, bridge, noise, and observable path end-to-end.
- Negative tests show that unsupported circuit sources, unsupported gates, fused
  blocks, or unsupported noise insertions fail before execution and name the
  first unsupported condition.
- The reproducibility bundle for Task 3 evidence records the circuit source
  type, ansatz, backend, lowered gate and noise schedule or equivalent auditable
  bridge metadata, plus unsupported-case diagnostics for negative evidence.
- Traceability target: satisfy the Phase 2 Task 3 evidence requirements in
  `DETAILED_PLANNING_PHASE_2.md`.
- Traceability target: satisfy the full-phase acceptance criterion requiring
  that the bridge is documented, partial by design, and usable for the anchor
  workflow.
- Traceability target: satisfy the bridge and support-matrix closure decisions
  recorded in `PRE_IMPLEMENTATION_COMPLETION_CHECKLIST.md` and the frozen bridge
  scope in `P2-ADR-011` and `P2-ADR-012`.

## Affected interfaces
- `qgd_Variational_Quantum_Eigensolver_Base.Generate_Circuit()` as the mandatory
  source of the guaranteed bridge path.
- The VQE-facing lowering or routing layer that carries the generated `HEA`
  circuit into density execution.
- `NoisyCircuit`, `GateOperation`, and `NoiseOperation` construction or adapter
  boundaries used by the supported density path.
- Pre-execution validation and error reporting for unsupported circuit, gate, or
  noise requests.
- The optional extension boundary for user-supplied `qgd_Circuit` or
  `Gates_block` inputs that lower fully inside the documented support matrix.
- Validation, benchmark, and reproducibility metadata that must record circuit
  source, bridge route, and unsupported-case diagnostics.
- Change classification: additive for the supported density-matrix VQE path, but
  stricter for ambiguous or unsupported manual-circuit requests, which become
  explicit hard failures rather than undocumented behavior.

## Publication relevance
- Supports Paper 1's claim that the density-matrix backend is integrated into
  SQUANDER's existing variational workflow rather than remaining only a
  standalone backend module.
- Makes bridge claims scientifically defensible by bounding them to a documented
  HEA-first support surface with auditable unsupported behavior.
- Provides contract evidence for the backend-integration, workflow-bridging, and
  reproducibility statements in the Phase 2 abstract, short paper, and full
  paper.
- Prevents the paper narrative from overstating circuit compatibility by clearly
  separating guaranteed HEA-first behavior from deferred full parity.
